# contest-report

用于撰写和迭代高校竞赛、课程设计、双创项目、训练营结项、申报答辩等场景中软件类作品设计开发文档的自定义Codexskill。

这个skill的目标不是把材料改写成企业PRD或SRS，而是把已有的技术报告、商业计划书、阶段报告、老师模板、代码仓库说明等材料，整理成更像正式比赛或项目提交文档的中文成稿。很多高校赛事和项目文档虽然名称不同，但在章节组织、图文配合、团队口吻、测试呈现和版式控制上有相近范式，这个仓库总结的正是这类写作规律。

## 适用场景

- 按比赛模板补写`需求分析`、`概要设计`、`详细设计`
- 将现有技术报告、商业计划书、阶段检查材料整合成一份参赛文档
- 对老师或组委会给出的 Word 模板做定向适配
- 在不丢掉项目真实细节的前提下，压缩内容长度并提升图文组织质量
- 将一套写作经验迁移到其他同类竞赛、课程项目或申报型文档
- 在已有阶段稿、技术报告和代码仓库之间做内容迁移，不把后期已写好的章节重新写空

## 这个 skill 重点解决什么

- 保持“我们作为项目团队提交作品”的叙述口吻，避免写成助手汇报
- 把比赛模板当作硬约束，而不是照搬企业文档目录
- 如果组委会给出明确格式要求，优先遵循其字体、字号、标题层级和提交格式
- 优先复用项目原始文档中的真实截图、架构图、流程图
- 在模板要求篇幅较短时，优先通过组合图、紧凑表格和总结段落提升信息密度，而不是直接把内容写空
- 在同时存在 AIC 技术报告和季度检查报告时，以较新的季度材料补充最近完成的功能与界面进展
- 测试报告优先写成带数据的自然叙述，而不是机械堆叠表格
- 安装与使用优先从真实代码仓库、README、settings和功能使用文档中反推

- `agents/openai.yaml`

## 当前写作偏好

- 主章节标题可使用`第一章 需求分析`、`第二章 概要设计`这类完整编号标题
- 主章节标题中的`第一章`等前缀和章节名本体应视为同一个标题整体，避免出现前缀和正文字体、字号不一致
- 主标题居中，分级标题左对齐，通常不超过三级，优先使用两级
- 如果赛事给出明确版式要求，则以赛事要求为准，而不是继续沿用此前个人偏好格式
- 图题和表题下置，正文必须显式引图引表
- 中文使用宋体，英文与数字使用 Times New Roman
- 避免 AI 味很重的`1. 2. 3.`流程分点
- 避免写成`一是、二是、三是`或`首先、然后、最后`堆起来的公式化段落
- 避免章节收束时写出像在和读者对话的句子
- `概要设计`不能只有架构图，平台核心界面截图和 WSI 标注截图应尽量纳入
- 渲染后如果出现换行只剩一两个汉字的怪行，需要回头改句子或缩图题
- 篇幅受限时，短小结尽量并入段尾，不单独立`本章小结`
- 表格列宽按信息密度分配，不默认所有列一样宽
- 表格过高时，要继续压单元格内边距、段落间距和措辞长度，而不是停留在只改列宽
- 表格尽量整页显示，题注也尽量和表格主体留在同一页
- `第七章 参考文献`等章节标题要保持结构顺序正确，不能因为程序插入顺序跑到正文后面

## 维护原则

- 真实比赛迭代中发现的新问题，先更新 `references/` 中的规则
- 只有工作流本身发生变化时，才修改 `SKILL.md`
- 优先做小步累积优化，而不是推翻重写
- 如果发现旧文档和新季度材料不一致，以当前实现状态为准

## 使用建议

如果你在 Codex 里调用这个 skill，可以从下面这些任务开始：

- “用`$contest-report`先重写需求分析”
- “用`$contest-report`按模板补概要设计”
- “用`$contest-report`把 AIC 技术报告和季度报告整合成比赛文档”

## 说明

这个仓库是面向真实项目迭代维护的 skill 仓库，不是静态模板仓库。它会随着比赛文档的实际打磨持续更新。
